# Persistence Layer

## Persistence Options

The engine itself does not require a database.

Several persistence strategies are possible:

### 1. Database Persistence (Recommended)

Skip to the **Overview** section of this document and you can use this document as an outline for database persistence.

To perform end to end testing, it is recommended to use a docker instance of the databases.

Ex. Mongo
```bash
sudo docker run -d --name mongo_tic_tac_db -p 27017:27017 mongo:latest
```
and verify that your .env lists mongo_tic_tac_db as the MONGO_DB and the URI points to localhost 27017

You can then use the following commands to verify the structure and insertion of a collection by entering the Mongo Shell.
```bash
sudo docker exec -it mongo_tic_tac_db mongosh
```
```javascript
show dbs
use mongo_tic_tac_db
show collections
db.simulation_experiences.find().pretty()
```

Ex. Postgress

Build a postgress container and verify that the port, user, password and DB are matching your env. 
```bash
sudo docker run -d \
  --name postgres_tic_tac \
  -e POSTGRES_USER=postgres_tic_tac_user \
  -e POSTGRES_PASSWORD=postgres_tic_tac_password \
  -e POSTGRES_DB=postgres_tic_tac_db \
  -p 5433:5432 \
  postgres:latest
```

Verify that the container exists, although it will be empty.
```bash
sudo docker exec -it postgres_tic_tac psql -U postgres_tic_tac_user -d postgres_tic_tac_db
```
```javascript
\dt
```
Build the tables from project root in terminal using init.sql
```bash
sudo docker exec -i postgres_tic_tac \
psql -U postgres_tic_tac_user -d postgres_tic_tac_db < persistence/postgres/initialization/init.sql
```


### 2. Pickle-Based Persistence (Simple Alternative)

For quick experimentation or local development, simulation data can also be stored using Python's `pickle` module.

Pickle allows Python objects to be serialized directly to disk without requiring a database.

Example:

```python
import pickle

with open("simulation.pkl", "wb") as f:
    pickle.dump(simulation_result, f)
```

Later the data can be restored:

```python
with open("simulation.pkl", "rb") as f:
    simulation_result = pickle.load(f)
```

This approach is useful for:

* quick experimentation
* local testing
* storing small datasets

However, pickle files are not suitable for large-scale datasets or analytics queries. 
For production-style workflows the database-backed persistence layer described below is recommended.  

## Overview

The `persistence` module is responsible for storing and retrieving the data generated by the simulation and training systems.
The project uses **two different databases**, each corresponding to the data which they are optimized for handling:

* **PostgreSQL** – relational data (games, moves, simulation runs, metadata)
* **MongoDB** – high-volume reinforcement learning experience data

Separating these allows the system to remain efficient and scalable while keeping the core game engine independent from storage details.

This persistence layer acts as a **boundary between the game/simulation systems and external persistent storage**.
The rest of the codebase does not directly interact with databases; instead it interacts with repository classes defined here.
The data is then pushed to the corresponding database from the connection files.

## Limitations

The major limitation of this database structure is that is will only hold data for 2 player simulations.  If you use the sim_engine with
more than 2 players, you cannot use the database to track these runs.

# Architecture Overview

Data flow through the system:

```
Simulation Engine
        │
        │ produces
        ▼
GameRunContext objects
        │
        │ aggregated into
        ▼
SimulationResult
        │
        ├──────────────► PostgreSQL (metadata + analytics)
        │
        └──────────────► MongoDB (training experiences)
```

This design allows:

* structured analytics queries on games
* scalable storage for reinforcement learning datasets
* independent scaling of the two storage systems

# PostgreSQL Layer

PostgreSQL stores the **structured relational data** about simulations and games.

Examples of data stored here include:

* simulation runs
* individual games
* moves within games
* player metadata
* experiment configurations

These datasets are relatively structured and benefit from relational querying.

The PostgreSQL layer is implemented in:

```
persistence/postgres
```

## postgres_connection.py

Responsible for:

* establishing database connections
* managing connection pooling
* exposing a reusable connection interface

All repository classes should obtain their database connection through this module.

---


# MongoDB Layer

MongoDB stores **reinforcement learning experience data**.

These datasets can grow extremely large and often contain nested structures such as tensors or vectors.

MongoDB is well suited for this type of data because:

* schema is flexible
* nested structures are natural
* high write throughput is supported

The MongoDB layer is implemented in:

```
persistence/mongo
```

# Database Overview

## PostgreSQL

Relational tables are used for structured metadata.

Conceptual schemas:


### Players

| Value          | SQL Type           | Python Type                  |
| -------------- | ------------------ | ---------------------------- |
| id             | Serial Primary Key | int                          |
| account_user   | Text               | str                          |
| player_type    | Text               | Literal["human", "computer"] |
| agent_name     | Text               | str / None                   |
| agent_version  | Text               | str / None                   |
| policy_name    | Text               | str / None                   |
| policy_version | Text               | str / None                   |
| created_at     | Timestamp          | datetime                     |

* Agent and policy info is None if human player

### Simulation Runs

| Value          | SQL Type           | Python Type    |
| -------------- | ------------------ | -------------- |
| id             | Serial Primary Key | int            |
| rule_set       | Text               | str            |
| encoder        | Text               | str            |
| reward_system  | Text               | str            |
| num_games      | Integer            | int            |
| created_at     | Timestamp          | datetime       |
| config_json    | JSONB              | dict[str, Any] |


### Games

| Value             | SQL Type                 | Python Type |
| ----------------- | ------------------------ | ----------- |
| id                | Serial Primary Key       | int         |
| simulation_run_id | Integer (FK sim_runs.id) | int         |
| player_one_id     | Integer (FK players.id)  | int         |
| player_two_id     | Integer (FK players.id)  | int         |
| winner_player_id  | Integer (FK players.id)  | int / None  |
| winner_in_game_id | Integer                  | int / None  |
| is_draw           | Boolean                  | bool        |
| total_moves       | Integer                  | int         |
| created_at        | Timestamp                | datetime    |


* If is_draw is true, winner_player_id = None


### Moves 

| Value             | SQL Type                | Python Type    |
| ----------------- | ----------------------- | -------------- |
| id                | Serial Primary Key      | int            |
| simulation_run_id | Integer (FK sim_runs.id)| int            |
| game_id           | Integer (FK games.id)   | int            |
| player_id         | Integer (FK players.id) | int            |
| player_id_in_game | Integer                 | int            |
| turn_number       | Integer                 | int            |
| row               | Integer                 | int            |
| col               | Integer                 | int            |
| reward            | Float                   | float          |
| board_state_json  | JSONB                   | list[list[]]   |



This structure allows efficient queries such as:

* win rate by agent
* average game length
* performance by policy
* replaying games move-by-move

To support efficient querying of training data, the following indexes are expected:

# Design Goals

The persistence layer follows several important principles:

### Separation of Concerns

The core engine and simulation code should remain completely independent from storage implementation.

Persistence logic lives only inside the `persistence` module.

---

### Repository Pattern

Database operations are abstracted through repository classes.

This allows:

* easier testing
* future storage changes
* cleaner application architecture

---

### Scalability

The dual-database design allows the system to scale naturally:

* PostgreSQL handles structured analytics
* MongoDB handles high-volume RL data

Each database can be independently optimized or scaled.

---

# Future Extensions

This module could later support:

* experiment tracking
* training dataset management

* distributed simulation pipelines
